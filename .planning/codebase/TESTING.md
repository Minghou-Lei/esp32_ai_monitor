# Testing

日期：2026-04-27

## 当前测试现状

当前仓库没有自动化测试框架，也没有单元测试、集成测试或 UI 测试。

当前能确认的验证方式只有：

- 本地 `ESP-IDF` 编译
- 上板烧录与串口观察

## 当前阶段最现实的验证方式

对于这类板卡工程，早期验证重点不在单元测试，而在 bring-up 验证：

1. `idf.py build`
2. `idf.py flash`
3. `idf.py monitor`
4. 观察硬件实际行为

## 板卡相关的关键验收项

### 显示

- 屏幕是否点亮
- 刷新是否稳定
- 是否出现花屏、撕裂、卡死

### 触摸

- 是否能上报点击
- 坐标是否正确
- 旋转方向是否匹配 UI

### 网络

- 是否能成功联网
- 断网后是否能重连
- 异常时 UI 是否有提示

### 业务协议

- 后端状态是否能正确解析
- 空数据、错误数据、超时数据是否有明确处理

## 当前缺口

- 没有自动化测试目录
- 没有 mock 后端
- 没有最小上板验收脚本
- 没有约定“每个功能改动必须怎么验证”

## 建议的近期测试策略

### 阶段 1

- 以 `build + flash + monitor + 屏幕观察` 为主

### 阶段 2

- 增加最小本地模拟后端，用于状态协议调试

### 阶段 3

- 为协议解析和状态存储补最小单元测试

## 特别说明

在这个项目里：

- “编译通过”不等于“功能可用”
- 涉及屏幕、触摸、网络的改动，必须经过真实板上验证
